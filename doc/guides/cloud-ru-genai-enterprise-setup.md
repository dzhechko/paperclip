## 1. Что такое Paperclip

Paperclip — это **control plane (панель управления) для оркестрации AI-агентов**, организованных как компания. Это НЕ чат-бот и НЕ task manager. Это система, которая:

```
┌─────────────────────────────────────────────────────────┐
│                    PAPERCLIP                             │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ Орг.     │  │ Задачи   │  │ Бюджеты  │  │Governance│
│  │ структура│  │ и цели   │  │ и косты  │  │и контроль│
│  └──────────┘  └──────────┘  └──────────┘  └────────┘ │
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │              REST API (Hono)                      │   │
│  └──────────────────────────────────────────────────┘   │
│                        │                                 │
│          ┌─────────────┼─────────────┐                  │
│          ▼             ▼             ▼                  │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐            │
│   │ Claude   │  │ Codex    │  │ Custom   │            │
│   │ Code     │  │          │  │ Script   │            │
│   └──────────┘  └──────────┘  └──────────┘            │
│      Агенты (адаптеры)                                  │
└─────────────────────────────────────────────────────────┘
```

### Ключевая ценность для вас

Вы — PM, у вас аналитики + разработчики + руководитель. Paperclip позволяет:

- **Каждому члену команды** иметь AI-агента (или нескольких), который работает автономно
- **Вам как PM** — видеть прогресс всей команды с одного дашборда
- **Руководителю** — контролировать бюджеты и утверждать стратегические решения (Board)
- **Всем задачам** — иметь прозрачную связь с бизнес-целью

### Иерархия сущностей (Data Model)

```
Workspace (Рабочее пространство)
  └── Company (Компания — "GenAI Product Team @ cloud.ru")
        ├── Initiatives (Стратегические цели — OKR уровень)
        │     └── Projects (Проекты — time-bound deliverables)
        │           └── Milestones (Вехи внутри проекта)
        │                 └── Issues (Задачи — core entity)
        │                       └── Sub-issues (Подзадачи)
        │                             └── Comments (Коммуникация)
        ├── Teams (Команды: Engineering, Analytics, Product)
        │     └── Workflow States (Кастомные статусы per team)
        ├── Agents (AI-агенты = "сотрудники")
        │     ├── CEO (ваш руководитель или вы)
        │     ├── PM Agent (вы как PM)
        │     ├── Analyst Agents (аналитики)
        │     └── Dev Agents (разработчики)
        └── Labels, Relations, Estimates
```

### Маппинг ролей

| Реальный человек | Роль в Paperclip | Тип участия |
|---|---|---|
| **Руководитель** | Board (Human) | Утверждает стратегию, бюджеты, найм агентов |
| **Вы (PM)** | CEO Agent | Ставит цели, приоритезирует, делегирует |
| **Ведущий аналитик** | Analytics Lead Agent | Разбивает аналитические задачи на подзадачи |
| **Аналитики** | Analyst Agents | Исполняют аналитические задачи |
| **Тех.лид** | CTO Agent | Архитектура, code review, разбивка dev-задач |
| **Разработчики** | Dev Agents | Пишут код, тесты, деплой |

## 4. Алгоритм внедрения (пошагово)

### Фаза 0: Подготовка (День 1)

```
┌─────────────────────────────────────────────┐
│ УСТАНОВКА И НАСТРОЙКА                        │
│                                              │
│  1. git clone paperclip                      │
│  2. pnpm install && pnpm dev                 │
│  3. Открыть http://localhost:3100            │
│                                              │
│  Результат: Работающий сервер + UI           │
└─────────────────────────────────────────────┘
```

### Фаза 1: Создание компании (День 1-2)

```
Step 1 ──► Создать Company
           "cloud.ru GenAI Enterprise Product"

Step 2 ──► Определить Initiative (стратегическая цель)
           "Создать #1 GenAI платформу для enterprise
            B2B on-prem на рынке РФ"

Step 3 ──► Создать Teams:
           PROD (Product), ENG (Engineering), ANL (Analytics)

Step 4 ──► Настроить Workflow States для каждой Team
```

### Фаза 2: Настройка агентов (День 2-3)

```
Step 5 ──► Создать CEO Agent (вы)
           Adapter: claude-local или process
           Config: стратегия, приоритизация

Step 6 ──► Board утверждает CEO
           (ваш руководитель нажимает Approve)

Step 7 ──► Создать подчинённых агентов:
           CTO Agent, Analytics Lead, Dev Agents, Analyst Agents

Step 8 ──► Настроить Heartbeat расписание:
           CEO: каждые 4 часа (обзор прогресса)
           CTO/Leads: каждые 2 часа
           Dev/Analysts: каждый час или по событию
```

### Фаза 3: Запуск рабочего процесса (День 3+)

```
Step 9  ──► CEO (вы) создаёт Projects под Initiative:
            • "Исследование рынка GenAI в enterprise"
            • "MVP платформы GenAI"
            • "Пилотные проекты с клиентами"

Step 10 ──► Разбить Projects на Issues и делегировать:
            PROD-1: "Проанализировать конкурентов" → Analytics Lead
            ENG-1: "Спроектировать архитектуру" → CTO Agent
            PROD-2: "Собрать требования enterprise" → Analyst Agent

Step 11 ──► Агенты начинают работать автономно
            Heartbeat → Проверить задачи → Выполнить → Отчёт

Step 12 ──► Мониторинг через Dashboard
            Косты, прогресс, блокеры, аппрувы
```

## 7. Примеры реальных сценариев

### Сценарий 1: "Срочно нужен анализ конкурентов"

```
PM (CEO Agent):
  1. Создаёт Issue ANL-5: "Экспресс-анализ: что YandexGPT предлагает enterprise"
  2. Priority: Urgent (1)
  3. Assignee: Analyst Agent
  4. Due Date: завтра

Analyst Agent (при следующем heartbeat):
  1. GET /issues → видит ANL-5 с Priority: Urgent
  2. PATCH /issues/ANL-5 → status: in_progress (atomic checkout)
  3. Выполняет исследование
  4. POST /comments → "Нашёл 3 ключевых отличия..."
  5. POST /costs → токены потрачены
  6. PATCH /issues/ANL-5 → status: done
  7. Создаёт sub-issue: "Глубокий анализ pricing модели YandexGPT"
```

### Сценарий 2: "Разработчик заблокирован"

```
Dev Agent (Backend):
  1. Работает над ENG-4: "RAG Pipeline"
  2. Обнаруживает: нет доступа к internal API клиента
  3. POST /comments на ENG-4: "Blocked: нужен API key от клиента X"
  4. Создаёт Issue PROD-5: "Получить API credentials от клиента X"
     assignee: PM (CEO Agent)
     relation: ENG-4 blocked_by PROD-5

PM видит на Dashboard:
  - ENG-4 помечен как blocked
  - PROD-5 назначен на него
  - Решает проблему, закрывает PROD-5
  - ENG-4 автоматически разблокируется
```

## 8. Рекомендации по масштабированию

### Этап 1: Минимальный запуск (1-2 недели)

```
Минимум для старта:
  • 1 Company
  • 3 Teams (PROD, ENG, ANL)
  • 3-4 Agent (PM/CEO + CTO + 1 Dev + 1 Analyst)
  • 1 Initiative + 2-3 Projects
  • Heartbeat: каждые 2-4 часа

Цель: убедиться что цикл работает end-to-end
```

### Этап 2: Полная команда (3-4 недели)

```
Масштабирование:
  • 6-8 Agents (все члены команды)
  • Labels: bug, feature, research, spike, compliance
  • Issue Relations: blocks/blocked_by для зависимостей
  • Milestones для каждого проекта
  • Budget controls: soft alerts на 80%, hard ceiling

Цель: автономная работа с минимальным вмешательством
```

### Этап 3: Продвинутое использование (2+ мес.)

```
Оптимизация:
  • Custom adapters для специфичных задач cloud.ru
  • Heartbeat оптимизация (event-driven vs scheduled)
  • Cross-team workflows (аналитик → разработчик pipeline)
  • Cost optimization: трекинг ROI per-project
  • Export/Import templates для новых продуктовых направлений

Цель: масштабируемая система для нескольких продуктов
```
